# 《瞬破寰宇》UI工业化逐步执行清单 v1.1

关联规范：`docs/瞬破寰宇-UI工业化规范-v1.1.md`  
执行模式：现有工程增量演进  
主验证场景：`assets/scenes/Phase10Main.scene`

---

## 1. 使用规则

- 必须按阶段顺序执行。
- 每一阶段只处理该阶段定义的范围。
- 上一阶段验收不通过，不进入下一阶段。
- 不恢复 `Phase8Main.scene` 或 `Phase10Main-Clean.scene`。
- 不覆盖用户已有未提交修改。
- 不修改现有公共接口、数据结构和配置格式。
- UI 代码修改必须先归类到 Step12 SOP，再进入实现。
- 每次代码修改后同时执行日志验收和截图验收。

状态标记：

```text
[ ] 未开始
[-] 进行中
[x] 已通过
[!] 阻塞
```

## 3. 阶段 1：视觉令牌与资源规则

目标：先统一规则，再改具体页面。

### 3.1 实施项

- [ ] 查找是否已有 UI 主题或样式配置。
- [ ] 若已有，扩展现有配置；若没有，再新增最小主题配置。
- [ ] 定义核心颜色令牌。
- [ ] 定义字体层级令牌。
- [ ] 定义间距、按钮尺寸和面板内边距令牌。
- [ ] 定义按钮状态颜色和动效参数。
- [ ] 定义现有四级装备品质映射。
- [ ] 定义图标资源映射规则。
- [ ] 明确资源是否需要动态加载，再决定是否进入 `resources`。

### 3.2 禁止项

- [ ] 不在多个 Panel 脚本中复制颜色和动效数值。
- [ ] 不为了统一目录批量移动现有 Prefab。
- [ ] 不在未改正式配置前增加绿色或红色装备品质。
- [ ] 不一次性重做全部 UI 美术资源。

### 3.3 验收

- [ ] 同一语义只有一个主题令牌来源。
- [ ] 品质样式由配置品质驱动。
- [ ] UI 文案和图标路径不散落硬编码。
- [ ] 主题配置加载失败时有明确错误处理。

## 4. 阶段 2：UIRoot、Canvas 与 Step12F 稳定性

目标：确认工业化改造建立在稳定首帧链路上。

### 4.1 SOP 分类

```text
首帧不显示       → SOP-UI-05
必须交互才刷新   → SOP-UI-06
Layout 不刷新    → SOP-UI-02
Render 延迟      → SOP-UI-03
Prefab 未初始化  → SOP-UI-01
```

### 4.2 实施项

- [ ] 检查 Canvas 设计分辨率为 720 × 1280。
- [ ] 检查 UIRoot 层级和安全区容器。
- [ ] 检查 Frame-0 全量布局和强制渲染链路。
- [ ] 检查首帧不依赖 dirtyMap 或用户交互。
- [ ] 不改变 `Phase10Main.scene` 中无关节点和 UUID。

### 4.3 日志验收

- [ ] 出现 `START` 或 `START_FALLBACK`。
- [ ] 出现 `PREFAB_INIT`。
- [ ] 出现 `LAYOUT_COMPUTE`。
- [ ] 出现 `FRAME_0_FORCE_FLUSH`。
- [ ] 出现 `RENDER_FLUSH`。
- [ ] 无重复 UUID 红字。
- [ ] 无 `Maximum call stack size exceeded`。
- [ ] `UIRoot` 的 LEVEL_1 黄字若存在，记录但不误判为阻塞。

### 4.4 截图验收

- [ ] 首帧无需点击、拖动或缩放即可显示。
- [ ] 无巨大灰块、错位、裁剪和遮挡。
- [ ] 关闭按钮和主要操作区可见。

### 4.5 通过条件

只有日志链路与首帧截图同时通过，才进入组件改造。

## 5. 阶段 3：BasePanel 与 Button 统一

目标：复用现有基类，统一生命周期和基础交互。

### 5.1 BasePanel 审计

- [ ] 确认现有 `show / hide / close / isShowing` 行为。
- [ ] 确认事件监听在销毁时完整清理。
- [ ] 确认安全区入口可被统一调用。
- [ ] 确认打开、关闭动画不会破坏 Step12 布局结果。
- [ ] 确认调试日志不会长期以错误级别污染控制台。

### 5.2 Button 审计与增强

- [ ] 搜索现有按钮反馈实现。
- [ ] 能复用时不新增组件。
- [ ] 无等价实现时，新增一个最小统一反馈组件。
- [ ] 支持 Normal、Pressed、Disabled。
- [ ] 按需支持 Selected、Loading。
- [ ] 点击反馈结束后恢复原始缩放和颜色。
- [ ] 危险操作使用 Danger 语义并保留确认流程。

### 5.3 验收

- [ ] 连续快速点击不会累计缩放或重复提交。
- [ ] Disabled 状态不能触发业务操作。
- [ ] 动效参数来自统一主题配置。
- [ ] 触控区域符合不小于 80px 的项目规范。

## 6. 阶段 4：UIManager 增量接管

目标：在不破坏现有模块 Manager 的前提下建立统一入口。

### 6.1 设计确认

- [ ] 定义 Panel 标识和注册方式。
- [ ] 定义无参数与带参数的打开方式。
- [ ] 定义页面、模态弹窗、Popup、Guide、Toast 层级。
- [ ] 定义重复打开策略。
- [ ] 定义返回键关闭顶部页面策略。
- [ ] 定义异步加载、加载失败和取消策略。
- [ ] 定义安全区广播方式。

### 6.2 必备入口

- [ ] `open(panelName)`。
- [ ] `close(panelName)`。
- [ ] `toggle(panelName)`。

### 6.3 增量接管顺序

- [ ] 先接管一个无参数、低风险页面。
- [ ] 再接管一个带参数页面。
- [ ] 再接管一个模态弹窗。
- [ ] 验证后才扩展到装备三面板。
- [ ] `Phase8UIManager` 和 `OperationsUIManager` 保持模块协调职责。

### 6.4 禁止项

- [ ] UIManager 不直接处理装备、背包、奖励或存档业务。
- [ ] 不修改现有 Panel 的公共 `open` 签名。
- [ ] 不一次性替换所有节点显隐调用。
- [ ] 不建立第二套 EventManager 或资源加载系统。

### 6.5 验收

- [ ] 同一 Panel 不会被重复实例化。
- [ ] 弹窗栈顺序正确。
- [ ] 模态遮罩能拦截下层输入。
- [ ] 返回键只关闭顶部允许关闭的页面。
- [ ] 加载失败不会留下遮罩或输入锁。

## 7. 阶段 5：TopBar、Icon 与 ItemSlot

目标：建立首批可量产的视觉组件。

### 7.1 TopBar

- [ ] 复用或增强现有 `EquipmentResourceBar`，先判断是否需要通用 TopBar。
- [ ] 显示灵石、玄铁晶、紫晶。
- [ ] 资源变化通过事件、Presenter 或现有数据流刷新。
- [ ] 适配顶部安全区。
- [ ] 确认 TopBar 不遮挡 Popup、Guide 和 Toast。

### 7.2 Icon

- [ ] 建立资源语义到 SpriteFrame 的统一映射。
- [ ] 检查小尺寸辨识度。
- [ ] 检查灵石、紫晶、玄铁晶风格一致。
- [ ] 优先合图，检查透明留白和纹理尺寸。

### 7.3 ItemSlot

- [ ] 优先增强现有 `EquipmentSlotItem` 和 `EquipmentItemView`。
- [ ] 支持 Frame、Icon、Level、Star。
- [ ] 支持 Selected、Locked、Equipped 等必要状态。
- [ ] 品质颜色只读取现有配置映射。
- [ ] 列表高频创建时检查对象池策略。

### 7.4 资源飞入

- [ ] 动画目标绑定 TopBar 对应资源节点。
- [ ] 动画只负责表现，最终数值以数据层为准。
- [ ] 高频播放时不产生明显 GC 或节点泄漏。
- [ ] 页面关闭或切场景时能安全取消。

### 7.5 验收

- [ ] 三种资源一眼可区分。
- [ ] 四级装备品质映射一致。
- [ ] ItemSlot 在背包和装备位复用时无样式漂移。
- [ ] TopBar 在刘海屏、小屏和弹窗场景下层级正确。

## 10. 阶段 8：新增页面扩展

目标：在样板稳定后按产品优先级扩展，而不是一次性铺开。

### 10.1 优先级判断

每个新页面必须先回答：

1. 是否帮助当前 MVP 或核心循环？
2. 是否有明确入口、出口和数据来源？
3. 是否能复用现有组件？
4. 是否适合个人开发和微信小游戏？
5. 是否需要现在实现，而不是仅预留？

### 10.2 建议顺序

```text
角色
通用背包
抽卡
活动
商城
宗门（V2，仅在正式立项后）
```

### 10.3 禁止项

- [ ] 不为 15 个角色写死节点数量。
- [ ] 不为单个活动创建不可复用的独立体系。
- [ ] 不把主角固定职业选择加入角色 UI。
- [ ] 不在未有业务系统时先造完整空壳页面。

## 13. 立即停止条件

出现以下任一情况，停止扩展并先修复基线：

```text
重复 UUID
Maximum call stack size exceeded
首帧必须交互后才显示
PrefabInit / LayoutCompute / RenderFlush 链路缺失
分解一件装备影响其他 uniqueId
新 UI 绕过 Manager/System 直接改 Data
为了视觉统一准备大规模重命名或目录迁移
```

---

## 14. 推荐首个执行批次

用户确认进入代码阶段后，首批只执行：

```text
1. 阶段 0 基线审计
2. 输出复用 / 增强 / 新增矩阵
3. 确认主题令牌应复用还是新增
4. 确认 UIManager 最小接管样板
5. 不改业务、不批量换资源、不重建场景
```

完成首批审计并再次确认修改范围后，再进入实际组件代码修改。
